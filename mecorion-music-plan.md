# План развития Mecorion Music

## 1. Цель продукта

Mecorion Music должен стать самостоятельным музыкальным сервисом экосистемы
Mecorion, объединяющим:

- онлайн-каталог Mecorion;
- локальную музыку пользователя;
- плейлисты, избранное и историю прослушивания;
- качественный музыкальный плеер;
- персональные рекомендации;
- синхронизацию между устройствами;
- подкасты, аудиокниги и радио в будущем.

Цель заключается не в копировании интерфейса Spotify или Яндекс Музыки, а в
достижении сопоставимого набора возможностей с сохранением визуального языка
Mecorion.

## 2. Текущее состояние прототипа

Уже реализовано:

- главная страница Music;
- поиск по локальному демонстрационному каталогу;
- раздел избранной музыки;
- отображение плейлистов;
- очередь воспроизведения;
- история недавно прослушанных треков;
- Playbar с управлением воспроизведением;
- перемешивание и режимы повтора;
- регулировка громкости и отключение звука;
- полноэкранный Player Mode;
- интеграция Media Session;
- импорт папки с музыкой с компьютера;
- представление локальной музыки списком;
- представление локальной музыки в виде проводника;
- отдельные фильтры для онлайн-каталога, локальной музыки, избранного и
  плейлистов;
- сохранение части пользовательских настроек в `localStorage`.

Текущая версия является frontend-прототипом. Для полноценного сервиса
необходимы постоянное хранилище, backend API и серверная обработка аудио.

## 3. Ближайший этап: полноценная медиатека

### 3.1. Модель музыкального каталога

Создать нормализованные сущности:

- `Track`;
- `Artist`;
- `Album`;
- `Genre`;
- `Playlist`;
- `PlaylistTrack`;
- `LibraryEntry`;
- `PlaybackHistory`;
- `Lyrics`;
- `AudioSource`.

Трек должен содержать идентификаторы исполнителей и альбома, жанры, год
выпуска, продолжительность, качество, признак explicit-контента, обложку,
источник и сведения о доступности.

### 3.2. Страницы сущностей

Добавить:

- страницу исполнителя;
- страницу альбома;
- страницу жанра;
- страницу отдельного плейлиста;
- страницу истории прослушивания;
- страницу всех понравившихся треков.

### 3.3. Управление плейлистами

Реализовать:

- создание и удаление плейлиста;
- изменение названия, описания и обложки;
- добавление и удаление треков;
- изменение порядка треков через drag-and-drop;
- публичные и приватные плейлисты;
- копирование плейлиста;
- восстановление недавно удалённого плейлиста.

### 3.4. Продвинутые фильтры

Расширить существующую систему:

- поиск внутри текущего раздела;
- фильтры по нескольким исполнителям и жанрам;
- диапазон годов;
- диапазон длительности;
- качество и формат аудио;
- дата добавления;
- дата последнего прослушивания;
- количество прослушиваний;
- наличие текста песни;
- explicit-контент;
- скачанные и доступные офлайн треки;
- сохранение набора фильтров;
- умные плейлисты на основе сохранённых условий.

Критерий готовности: пользователь может найти нужный трек в большой медиатеке,
не используя глобальный поиск.

## 4. Тексты песен

### 4.1. Форматы

Поддержать:

- обычный несинхронизированный текст;
- синхронизированный текст по строкам;
- перевод;
- транслитерацию;
- несколько языков;
- локальные файлы `.lrc`;
- встроенные тексты из метаданных аудиофайла.

### 4.2. Модель данных

Сущность `Lyrics` должна хранить:

- идентификатор трека;
- язык;
- тип текста;
- источник;
- статус модерации;
- сведения о правах;
- версию;
- массив строк с `startMs`, `endMs` и `text`.

### 4.3. Интерфейс

Добавить в Player Mode:

- вкладки «Обложка», «Текст» и «Очередь»;
- автоматическую прокрутку синхронизированного текста;
- переход к позиции трека по нажатию на строку;
- настройку размера текста;
- отображение оригинала и перевода;
- полноэкранный режим текста.

### 4.4. Пользовательское добавление

Разделить два сценария:

- личный текст для локального использования;
- предложение текста для публикации в каталоге Mecorion.

Публичные тексты должны проходить модерацию и проверку прав.

## 5. Backend и постоянное хранение

### 5.1. Предлагаемый стек

- Node.js и NestJS для API;
- PostgreSQL для основных данных;
- Prisma для схемы и миграций;
- Redis для кеша, сессий и очередей;
- S3-совместимое хранилище для аудио и изображений;
- BullMQ для фоновых задач;
- FFmpeg для анализа и преобразования аудио;
- Meilisearch на первом этапе для полнотекстового поиска.

### 5.2. Основные API

Создать API для:

- каталога;
- поиска;
- медиатеки пользователя;
- плейлистов;
- лайков;
- очереди;
- истории прослушивания;
- текстов песен;
- пользовательских настроек;
- синхронизации устройств.

### 5.3. Локальный кеш

Использовать IndexedDB для:

- кеша каталога;
- состояния очереди;
- истории;
- локальных метаданных;
- обложек;
- подготовленных офлайн-треков.

Критерий готовности: данные пользователя сохраняются после перезагрузки и
доступны после входа на другом устройстве.

## 6. Аудиодвижок

Добавить:

- бесшовное воспроизведение;
- crossfade;
- нормализацию громкости;
- эквалайзер и пресеты;
- настройку качества;
- поддержку Lossless;
- автоматическое продолжение похожей музыкой;
- таймер сна;
- запоминание позиции;
- управление очередью через drag-and-drop;
- воспроизведение следующего трека без задержки;
- обработку ошибок сети и недоступных файлов.

Для аудиодвижка следует выделить отдельный слой, не зависящий от компонентов
Vue. UI должен отправлять команды проигрывателю, а не управлять `<audio>`
напрямую.

## 7. Офлайн-режим

Реализовать:

- скачивание трека, альбома или плейлиста;
- отображение прогресса загрузки;
- управление занятым местом;
- автоматическое удаление старого кеша;
- режим «только скачанное»;
- восстановление незавершённых загрузок;
- проверку срока доступа к лицензированному контенту.

Для полноценного офлайн-режима и доступа к файловой системе стоит рассмотреть
отдельное desktop-приложение на Tauri.

## 8. Сбор событий и аналитика прослушивания

До реализации рекомендаций необходимо собирать:

- начало воспроизведения;
- позицию пропуска;
- дослушивание;
- повтор;
- лайк и дизлайк;
- добавление в плейлист;
- удаление из медиатеки;
- поисковый запрос;
- переход к треку из рекомендации;
- выбранный источник музыки.

События должны отправляться пакетами и не блокировать воспроизведение.

## 9. Рекомендации

Развивать рекомендации поэтапно:

1. Редакторские подборки.
2. Рекомендации по жанрам и исполнителям.
3. Радио по треку, исполнителю и плейлисту.
4. Миксы на основе истории пользователя.
5. Совместная фильтрация по поведению пользователей.
6. Контекстные рекомендации по времени, активности и настроению.

Будущие разделы:

- «Моя волна»;
- «Для вас»;
- «Давно не слушали»;
- «Новые релизы»;
- «Похоже на любимое»;
- персональные дневные и недельные миксы.

AI-функции следует добавлять после накопления качественных событий
прослушивания. Иначе рекомендации будут случайными и плохо объяснимыми.

## 10. Синхронизация устройств

Добавить:

- единую очередь на всех устройствах;
- продолжение воспроизведения с прежней позиции;
- список активных устройств;
- переключение воспроизведения между устройствами;
- дистанционное управление плеером;
- синхронизацию настроек качества и громкости;
- разрешение конфликтов при одновременном редактировании плейлиста.

## 11. Социальные функции

После завершения основного плеера реализовать:

- публичные профили;
- подписки на пользователей и исполнителей;
- публикацию плейлистов;
- совместные плейлисты;
- приглашения по ссылке;
- групповые сессии прослушивания;
- общую очередь;
- отображение автора добавленного трека;
- настройку приватности активности.

Комментарии и личные сообщения не являются ранним приоритетом, поскольку
потребуют отдельной системы модерации.

## 12. Подкасты, аудиокниги и радио

Эти типы контента должны использовать общий аудиодвижок, но разные модели
интерфейса.

Для подкастов:

- RSS-импорт;
- эпизоды и сезоны;
- скорость воспроизведения;
- пропуск тишины;
- подписки;
- автоматическая загрузка новых эпизодов.

Для аудиокниг:

- главы;
- закладки;
- заметки;
- сохранение позиции;
- таймер сна.

Для радио:

- прямые эфиры;
- программная сетка;
- история прозвучавших треков;
- метаданные текущей передачи.

## 13. Права и модерация

Для публичного онлайн-каталога потребуются:

- учёт правообладателей;
- региональные ограничения;
- даты начала и завершения лицензии;
- обработка запросов на удаление;
- модерация обложек, метаданных и текстов;
- журнал изменений;
- расчёт статистики воспроизведений;
- отчётность для правообладателей;
- parental control и explicit-фильтры.

## 14. Качество продукта

Обязательные технические задачи:

- перевод Music на TypeScript;
- unit-тесты для store, фильтров и очереди;
- component-тесты для элементов плеера;
- end-to-end тесты основных пользовательских сценариев;
- обработка ошибок и понятные состояния загрузки;
- accessibility и управление с клавиатуры;
- локализация;
- мониторинг frontend и backend ошибок;
- метрики скорости запуска и начала воспроизведения;
- оптимизация размера frontend bundle.

## 15. Рекомендуемый порядок реализации

### Итерация 1. Медиатека

- нормализовать модели треков, альбомов и исполнителей;
- добавить страницы исполнителя и альбома;
- реализовать CRUD плейлистов;
- завершить продвинутые фильтры;
- перенести пользовательские данные в IndexedDB.

### Итерация 2. Тексты

- создать модель `Lyrics`;
- добавить парсер `.lrc`;
- читать встроенные тексты локальных файлов;
- добавить вкладку текста в Player Mode;
- реализовать синхронизацию строк с воспроизведением.

### Итерация 3. Backend

- создать API и базу данных;
- подключить авторизацию Mecorion;
- синхронизировать библиотеку, лайки и плейлисты;
- добавить загрузку аудио и обработку FFmpeg.

### Итерация 4. Качество воспроизведения

- переработать аудиодвижок;
- добавить gapless, crossfade и нормализацию;
- реализовать настройки качества;
- добавить устойчивое восстановление после ошибок.

### Итерация 5. Офлайн и устройства

- реализовать скачивание;
- добавить управление кешем;
- синхронизировать очередь и позицию;
- создать прототип удалённого управления устройством.

### Итерация 6. Рекомендации

- внедрить события прослушивания;
- создать простые рекомендации;
- добавить радио и персональные миксы;
- измерять качество рекомендаций.

### Итерация 7. Социальный слой

- публичные профили;
- совместные плейлисты;
- групповые сессии;
- настройки приватности.

## 16. Ближайшие задачи

Следующий практический набор работ:

1. Спроектировать TypeScript-интерфейсы музыкальных сущностей.
2. Добавить маршруты исполнителей и альбомов.
3. Реализовать создание и редактирование плейлистов.
4. Расширить фильтры диапазонами и сохранёнными наборами.
5. Создать модель текстов и `.lrc`-парсер.
6. Добавить текст песни в Player Mode.
7. Начать сохранять медиатеку и локальные метаданные в IndexedDB.
